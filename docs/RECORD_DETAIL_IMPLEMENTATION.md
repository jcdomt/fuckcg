## 数据流程

### 1. 获取运动记录列表
```
用户打开"运动记录"tab
    ↓
前端调用 Bridge.getSportRecords(query)
    ↓
Java 处理：
  - 验证登录凭据
  - 拼装查询参数 (xh, type)
  - 调用 http.get(/api/l/v7/sportlist)
  - 解析服务器响应
  - 标准化每条记录
  - 按时间从近到远排序
    ↓
返回标准化的 JSON 数据
    ↓
前端渲染列表卡片（带点击事件）
```

### 2. 点击查看详情
```
用户点击某条记录
    ↓
调用 RecordDetailModule.openDetail(record)
    ↓
将记录数据保存到 sessionStorage
    ↓
跳转到 record-detail.html
    ↓
详情页面从 sessionStorage 读取数据
    ↓
使用 createDetailCard() 函数生成 4 个信息卡片
    ↓
展示完整的运动详情
```

### 3. 返回列表
```
用户点击返回按钮或浏览器后退
    ↓
调用 RecordDetailModule.closeDetail()
    ↓
清空 sessionStorage
    ↓
返回列表页面 (window.history.back())
```

## 关键设计决策

### 1. 分离 HTML 文件
- ✅ 为什么：避免单个文件过大，职责清晰
- ✅ 工作流程：列表 (work.html) ↔ 详情 (record-detail.html)
- ✅ 数据传递：通过 sessionStorage 中间件

### 2. 独立 JS 模块
- ✅ 为什么：逻辑复用，易于测试和维护
- ✅ 模块名：RecordDetailModule（全局对象）
- ✅ 导出方法：openDetail, closeDetail, formatRecord 等

### 3. 后端处理关键数据
- ✅ 为什么：减少前端复杂度，统一数据格式
- ✅ 处理：normalizeRecord() 提取 13 个关键字段
- ✅ 排序：sortRecordsByTime() 按时间从新到旧

### 4. 样式独立维护
- ✅ 为什么：避免冲突，便于主题管理
- ✅ 方法：统一使用 CSS 变量 (--primary, --text 等)
- ✅ 响应式：媒体查询支持移动端
